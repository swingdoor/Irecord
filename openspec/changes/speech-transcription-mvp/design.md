## Context

这是语音转写助手项目的第一个 change（MVP），目标是快速验证技术可行性并交付可用的基础版本。

**当前状态**：
- 项目从零开始，无现有代码
- 已确定技术栈：Electron + React + Sherpa-ONNX
- 已选定核心模型：Qwen3-ASR-0.6B（阿里 2025 年最新模型）

**约束条件**：
- 仅支持 Windows 平台
- 仅支持 CPU 模式（GPU 支持在后续 change）
- 音频文件处理（非实时流式）
- 单一模型（多模型支持在后续 change）
- 10 分钟以内音频（长音频处理在后续 change）

**利益相关者**：
- 最终用户：需要本地化、隐私安全的语音转写工具
- 开发团队：需要稳定的基础架构供后续功能扩展

## Goals / Non-Goals

**Goals:**
- 交付可运行的 Electron 桌面应用
- 实现完整的语音识别流程（上传 → 处理 → 展示 → 导出）
- 集成 Sherpa-ONNX + Qwen3-ASR-0.6B，验证识别精度
- 提供基础但完整的用户体验
- 建立可扩展的代码架构，为后续功能打基础

**Non-Goals:**
- 说话人分离（Change 2）
- 标点预测和关键词提取（Change 3）
- 长音频和批量处理（Change 4）
- GPU 加速和多模型支持（Change 5）
- 高级导出格式和可视化（Change 6）
- 生产级打包和分发（Change 7）
- 实时流式识别
- 云端同步或协作功能

## Decisions

### 1. 架构模式：Electron 主进程 + Worker 线程

**决策**：使用 Worker 线程处理音频识别，避免阻塞主进程和 UI。

**理由**：
- Sherpa-ONNX 的 CPU 推理是计算密集型操作，会阻塞 Node.js 事件循环
- Worker 线程可以利用多核 CPU，提升性能
- 主进程负责 IPC 通信和任务调度，保持 UI 响应

**备选方案**：
- ❌ 子进程（child_process）：进程间通信开销大，数据序列化成本高
- ❌ 主进程直接处理：会冻结 UI，用户体验差

### 2. 前端框架：React + TailwindCSS

**决策**：使用 React 18 + TailwindCSS 3 构建 UI。

**理由**：
- React 生态成熟，组件化开发效率高
- TailwindCSS 快速构建现代化 UI，无需手写大量 CSS
- 与 Electron 集成良好，社区方案成熟

**备选方案**：
- ❌ Vue：团队对 React 更熟悉
- ❌ 原生 HTML/CSS：开发效率低，难以维护

### 3. 音频处理：FFmpeg + Sherpa-ONNX 内置重采样

**决策**：使用 FFmpeg 提取视频音轨和格式转换，Sherpa-ONNX 处理重采样。

**理由**：
- FFmpeg 是音视频处理的行业标准，支持所有主流格式
- Sherpa-ONNX 内置重采样功能，无需额外依赖
- 打包 FFmpeg 预编译二进制，用户无需安装

**备选方案**：
- ❌ 纯 Node.js 音频库（如 node-wav）：功能有限，不支持视频
- ❌ 要求用户安装 FFmpeg：增加使用门槛

### 4. 模型选择：Qwen3-ASR-0.6B

**决策**：MVP 阶段仅集成 Qwen3-ASR-0.6B 单一模型。

**理由**：
- 2025 年 1 月最新发布，代表最新技术水平
- 支持 52 种语言，中文精度优秀
- 模型大小适中（~600MB），平衡精度和体积
- Sherpa-ONNX 官方已支持

**备选方案**：
- ❌ SenseVoice-Small：功能更多（情感识别），但 MVP 不需要
- ❌ Paraformer-large-zh：纯中文场景略优，但多语言支持弱
- ❌ 同时集成多个模型：增加复杂度和安装包体积

### 5. 状态管理：Zustand

**决策**：使用 Zustand 进行前端状态管理。

**理由**：
- 轻量级（~1KB），学习曲线平缓
- API 简洁，无需 Provider 包裹
- 足够满足 MVP 的状态管理需求

**备选方案**：
- ❌ Redux：过于重量级，MVP 不需要
- ❌ Context API：跨组件通信繁琐

### 6. 构建工具：Vite

**决策**：使用 Vite 作为前端构建工具。

**理由**：
- 开发服务器启动快，HMR 体验好
- 与 Electron 集成方案成熟（electron-vite）
- 构建速度快

**备选方案**：
- ❌ Webpack：配置复杂，构建慢
- ❌ Create React App：不够灵活，已不推荐

### 7. 数据存储：本地文件系统

**决策**：MVP 阶段不使用数据库，处理结果直接导出为文件。

**理由**：
- 简化架构，快速交付
- 用户需求是"处理完导出"，不需要持久化存储
- 后续 change 可按需添加 SQLite

**备选方案**：
- ❌ SQLite：增加复杂度，MVP 不需要
- ❌ IndexedDB：浏览器存储，不适合桌面应用

## Risks / Trade-offs

### 1. 模型文件体积大（~600MB）
**风险**：安装包体积大，下载和安装时间长。

**缓解措施**：
- 首次启动时检查模型，提示用户下载（而非打包在安装包中）
- 提供下载进度显示
- 后续 change 优化为增量下载

### 2. CPU 模式性能有限
**风险**：处理速度慢，用户体验不佳。

**缓解措施**：
- MVP 限制音频长度（10 分钟以内）
- 显示预计剩余时间，管理用户预期
- Change 5 添加 GPU 加速

### 3. 仅支持 Windows
**风险**：限制用户群体。

**缓解措施**：
- Electron 和 Sherpa-ONNX 都是跨平台的，后续扩展到 macOS/Linux 成本低
- MVP 聚焦单一平台，快速验证

### 4. 无错误恢复机制
**风险**：处理失败后用户需要重新上传。

**缓解措施**：
- 基础错误处理和友好提示
- Change 4 添加断点续传和任务恢复

### 5. Sherpa-ONNX Node.js 绑定可能不稳定
**风险**：原生模块在 Windows 上可能有兼容性问题。

**缓解措施**：
- 使用官方预编译的 npm 包
- 充分测试不同 Windows 版本
- 准备降级方案（调用 CLI 版本）

## Migration Plan

**部署步骤**：
1. 开发环境搭建（Node.js, Electron, 依赖安装）
2. 实现核心功能（按 tasks.md 顺序）
3. 本地测试（不同音频格式、不同长度）
4. 打包为开发版本（electron-builder）
5. 内部测试和反馈收集

**回滚策略**：
- MVP 是全新项目，无需回滚
- 如果技术方案不可行，可切换到备选方案（如不同的 ASR 引擎）

## Open Questions

1. **模型下载策略**：首次启动下载 vs 打包在安装包？
   - 倾向：首次启动下载，减小安装包体积
   - 需要：实现下载进度和失败重试

2. **错误日志收集**：是否需要日志系统？
   - 倾向：添加基础日志（console + 文件），方便调试
   - 使用：loguru 或 winston

3. **配置文件格式**：JSON vs YAML？
   - 倾向：JSON，Node.js 原生支持
   - 位置：用户目录 `~/.speech-transcriber/config.json`

4. **临时文件清理**：何时清理音频预处理的临时文件？
   - 倾向：处理完成后立即清理
   - 失败情况：应用退出时清理
