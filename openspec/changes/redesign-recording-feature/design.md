## Context

当前录音功能存在以下问题：
- 录音页面采用居中式布局，浪费左右空间
- 录音记录与文件上传任务混在一起，数据模型不清晰
- 自动开始录音让用户措手不及
- 缺少暂停功能，用户无法临时中断录音
- 精准校对功能不够灵活

本次重设计需要在保持现有文件上传功能不变的前提下，重构录音功能的完整交互流程。

## Goals / Non-Goals

**Goals:**
- 提供清晰的信息架构，分离实时录音和文件上传
- 优化录音页面布局，充分利用屏幕空间
- 改进录音控制逻辑，支持手动开始和暂停
- 提供灵活的精准校对选项
- 复用现有组件，保持 UI 风格一致

**Non-Goals:**
- 不修改现有文件上传功能的核心逻辑
- 不改变音频识别引擎的实现
- 不涉及新的 AI 分析功能

## Decisions

### Decision 1: 独立的数据表
**选择**: 创建独立的 realtime_recordings 表，与 tasks 表分离

**理由**:
- 实时录音记录和文件分析任务是两种不同的业务实体
- 分离后数据模型更清晰，查询更高效
- 避免在 tasks 表中混入不同类型的记录

**替代方案**:
- 在 tasks 表中增加 type 字段区分：会导致表结构复杂，查询需要额外过滤

### Decision 2: 文件命名规则
**选择**: 使用 `语音_YYYYMMDDHHmmss.wav` 格式（14位数字）

**理由**:
- 保证文件名唯一性，避免冲突
- 按时间自然排序
- 不需要管理文件夹结构，简化实现

**替代方案**:
- 使用 UUID：不直观，无法从文件名看出创建时间
- 按月份分文件夹：增加管理复杂度

### Decision 3: 暂停期间不存储数据
**选择**: 暂停时断开音频输入，不存储也不识别音频数据

**理由**:
- 实现简单，避免缓存管理
- 用户预期：暂停即停止录音
- 节省存储空间和计算资源

**替代方案**:
- 缓存暂停期间的音频：增加复杂度，用户可能不需要

### Decision 4: 复用任务详情页组件
**选择**: 实时录音详情页复用 AudioPlayer、TranscriptPanel、AiPanel 组件

**理由**:
- 保持 UI 风格一致
- 减少代码重复
- 用户体验统一

**替代方案**:
- 创建新的组件：增加维护成本，UI 可能不一致

### Decision 5: Tab 分隔而非独立页面
**选择**: 在任务列表区域使用 Tab 分隔，而非创建独立的录音列表页面

**理由**:
- 保持主页面的完整性
- 用户可以快速切换查看不同类型的任务
- 减少页面跳转

**替代方案**:
- 独立的录音列表页面：增加导航层级，用户体验不佳

## Risks / Trade-offs

### Risk 1: 数据迁移
**风险**: 现有的录音记录（如果有）需要迁移到新表

**缓解**: 
- 检查现有数据库中是否有 status='recording' 的任务
- 提供迁移脚本将这些记录转移到 realtime_recordings 表

### Risk 2: 文件共享
**风险**: 实时录音记录和精准校对任务共享同一个 WAV 文件，删除时需要注意

**缓解**:
- 删除录音记录时，检查是否有关联的任务正在使用该文件
- 如果有关联任务，只删除数据库记录，保留文件

### Risk 3: 暂停功能的用户理解
**风险**: 用户可能不理解暂停期间的音频会被丢弃

**缓解**:
- 在 UI 上明确显示暂停状态
- 暂停时显示提示："暂停期间的音频不会被录制"

### Risk 4: 组件复用的灵活性
**风险**: 复用组件可能导致实时录音详情页无法独立定制

**缓解**:
- 通过 props 传递不同的配置
- 如果需要特殊定制，可以创建包装组件

## Migration Plan

### Phase 1: 数据库迁移
1. 创建 realtime_recordings 表
2. 检查现有 tasks 表中是否有 status='recording' 的记录
3. 如果有，迁移到 realtime_recordings 表

### Phase 2: 后端实现
1. 实现 realtime_recordings 表的 CRUD 操作
2. 修改录音相关的 IPC handlers
3. 实现精准校对任务创建逻辑

### Phase 3: 前端实现
1. 修改 TaskListPage，添加 Tab 切换
2. 创建 RealtimeRecordingTable 组件
3. 重构 RecordingPage 布局和控制逻辑
4. 创建 RealtimeRecordingDetailPage
5. 实现确认对话框

### Phase 4: 测试和优化
1. 测试录音控制流程
2. 测试精准校对功能
3. 测试数据持久化
4. 优化 UI 细节

### Rollback Strategy
如果出现严重问题，可以：
1. 回滚数据库迁移（保留 realtime_recordings 表，但不使用）
2. 恢复原有的录音页面和逻辑
3. 用户数据不会丢失（WAV 文件和识别结果都已保存）

## Open Questions

1. **删除策略**: 删除录音记录时，是否同时删除 WAV 文件？
   - 建议：提供选项让用户选择

2. **精准校对模型**: 默认使用哪个模型进行精准校对？
   - 建议：使用用户在设置中配置的默认模型

3. **录音记录的保留期限**: 是否需要自动清理旧的录音记录？
   - 建议：暂不实现，由用户手动管理
